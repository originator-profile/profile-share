# OP とは

OP とは、Web コンテンツの**対象範囲を特定**し、その作成者・管理者の**基本情報**と**資格情報**を検証可能な形で付与し、Web の後方互換性を維持しつつ信頼性と透明性を高める技術です。

Web ページはさまざまな組織が作ったコンテンツが混在した状態で提示されます。 OP を使えば個々のコンテンツの**範囲を特定**し、それぞれの出所についての情報を紐づけることができます。コンテンツ・サイトの情報は **[Document Profile (DP)](/terminology/document-profile)**、その発信者に関する情報は **[Originator Profile (OP)](/terminology/originator-profile.md)** として定義されます。

TLS 等での証明書の標準として広く普及している [X.509](https://www.itu.int/rec/T-REC-X.509) とは異なり、 OP では組織の ID にデフォルトで信頼性を表す情報として**資格情報**が付与され、 OP に検証可能な形で同梱されます。この資格情報は、コンテンツ閲覧者がコンテンツ発行組織の信頼性を判断するのに役立ちます。資格情報の質は OP 技術やレジストリを運営する**ガバナンス設計**によって担保されます。

OPは OP ID の発行について基準を設けます。 発行を希望する組織についての**基本情報**を第三者による確認をした上で OP レジストリに登録し、検証可能なデータとして OP に記載します。発行の前に組織の実在性確認などの基本的な確認を受けるため、 OP ID を保有していること自体が信頼性のベースラインとして使えます。発行基準はガバナンスにより決まり、 Web の信頼性・透明性を高めるようなガバナンス設計をします。

OP/DP はデータフォーマットとして Verifiable Credential (VC) を採用しています。 VC は、 IHV モデルと呼ばれる VC を利用する人の役割分担を発行者、保有者、検証者の3者で考えるモデルに則っています。このモデルによりプライバシーの保護や検証時のサーバー負荷軽減などのメリットを享受できます。

OP はインターネットの常識や設計を参考にし、分散型のアーキテクチャで社会実装されていくことを想定しています。これにより特定のレジストリ等の運営団体に依存しない技術になることを最初から目指しています。

従来のインターネットでは、ユーザーが、閲覧しているコンテンツがどのような組織によって作成され流通しているのかを確認することができませんでした。

コンテンツよりも粗い粒度での確認に関しては、EV証明書が利用できるでしょう。Web サイトのドメインを保有している法人の確認には、 EV証明書が利用できます。 EV証明書は発行の際に証明書取得者の本人性を確認することが義務付けられています。EV証明書が設置されている Web サイトを HTTPS を使って訪れたユーザーは、ブラウザのアドレスバーから証明書の情報を確認することで、ドメイン保有者の実在性が確認できました。一方で、 EV証明書はユーザーの被害を防止するためには効果が薄く、ブラウザベンダーの支持を得られず、より取得要件の緩い OV証明書や DV証明書と同じような扱いになりました。また、 EV証明書ではドメイン保有組織が実在しているかどうかを確認できるにとどまり、その組織が健全なサイト運営者、コンテンツ提供者かどうかを知ることはできません。OP はドメイン単位の検証にとどまらず、ドメインとコンテンツの関係性を考慮した設計になっています。また、組織の確認により高い基準を設けます。

あるドメインのサーバーのリソース群を別のドメインから再配信可能にする技術として、 [AMP](https://amp.dev/), [Signed HTTP Exchange (SXG)](https://wicg.github.io/webpackage/draft-yasskin-http-origin-signed-responses.html), [Web Packaging](https://github.com/WICG/webpackage/) がありました。リソースに対して元のドメインの証明書で署名することによって再配信を可能にする技術でしたが、ドメイン単位でしか署名ができないため上記の証明書や TLS の課題が残ります。そもそも別サーバーからの再配信のための技術であり、コンテンツや発行者の信頼性に関する技術ではありません。

また、従来の Web においてページやコンテンツの情報をデジタル化する仕組みは [Open Graph Protocol (OGP)](https://ogp.me/) や [JSON-LD](https://json-ld.org/) [構造化データ](https://developers.google.com/search/docs/appearance/structured-data/intro-structured-data) のようなものがありましたが、記載情報に対して発信者の署名がなされている仕組みではなく、発信者に関する信頼性を機械的に検証可能なデータ及び実装ではありませんでした。

その他の類似技術との関係については[よくある質問](/faq.md#類似技術との比較)を参照してください。

これらの特徴により、 OP 技術は Web コンテンツを閲覧するユーザーに次のような恩恵を与えます:

- サイトやコンテンツの出所を知ることができます。発信者は Web コンテンツに紐づく DP の署名者であり、その情報は OP から得られます。 OP は、 X.509 証明書とは異なり、ID にデフォルトで信頼性情報が同梱されます。そのためサイトやコンテンツの信頼性をよりよく理解することができます。
- 情報の真正性が担保されます。 DP に記載の情報は署名によって真正性が担保された情報であり、ユーザーはそれらの情報が正しい発信者によって書かれたものであることを OP 拡張機能を利用して確証できます。この真正性は OGP を利用してメタデータをつけることでは実現できません。
- コンテンツの信頼性を確認できます。資格情報は組織の信頼性の要約として機能します。ユーザーは資格情報を見ることで組織の信頼性を容易に判断することができます。また、資格情報は OP レジストリ運用者とは独立した第三者による認証なため、信頼性を中立な立場から判断することができます。
- Web コンテンツの流通を確認できます。Web ページはさまざまな組織が作ったコンテンツが混在した状態で提示されます。コンテンツは複数のサイトに掲載されたり、別のサイトに再配信されたりします。 OP を使うことで、ページ単位、ドメイン単位での信頼性だけでなく、コンテンツ単位での信頼性の判断や、流通元、出所を確認することができます。
